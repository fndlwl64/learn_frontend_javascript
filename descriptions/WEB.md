### Web Browser
- 웹 서버에서 HTTP를 통해 양방향 통신
- HTML 문서 및 자원 등의 컨텐츠 제공
- GUI 기반의 소프트웨 프로그램

---

### Web의 동작원리
#### Web Browser의 동작 방식
- 요청과 응답(request, response)
    - 사용자가 선택한 자원을 서버에 요청하고 서버의 응답을 브라우저에 출력(Rendering)하는 방식으로 동작
    - > Tip - URI(Uniforme Resource Identifier) 형식으로 요청 주소 표시

#### Web Browser의 동작 과정
1. 사용자가 URL 입력
2. DNS 서버가 도메인 -> IP 주소 변환
3. HTTP 요청 메시지 생성 -> TCP를 통해 서버 전송
4. 서버가 요청 처리 후 HTTP 응답 메시지 반환
5. 브라우저가 응답을 렌더링(Rendering)하여 화면 출력
- > Request -> Response -> Rendering
- 참고 : [동작 상세](./WEB_Section01.md)

### Browser의 기본 구조와 특징
#### 기본 구조
- 사용자 인터페이스 > 브라우저 엔진 
    - 브라우저 엔진 > 자료저장소
    - 브라우저 엔진 > 렌더링 엔진
      - 렌더링 엔진 > 자바스크립트 해석시
      - 렌더링 엔진 > UI 벡엔드
#### 사용자 인터페이스(UI)
- 화면에 출력되어 사용자에게 직접 보여주는 영역
#### 브라우저 엔진(Browser Engine)
  - 사용자 인터페이스와 렌더링 엔진 사이의 동작을 제어
  - 네트워크 요청, 보안 정책, 쿠키/캐시 관리 등 브라우저 전반을 조율
#### 렌더링 엔진(Rendering Engine)
  - HTML, CSS, JavaScript 등을 파싱하여 DOM, CSSOM 트리를 생성
  - 레이아웃(Layout)과 페인팅(Painting) 과정을 거쳐 최종적으로 화면에 출력
- > Tip: 흔히 브라우저 엔진과 렌더링 엔진을 혼용하기도 하지만, 실제로는 역할이 구분됨

#### 자바스크립트 해석기(JavaScript Interpreter)
- 자바스크립트 코드를 해석하고 실행하는 역할
##### 구글 크롭의 V8엔진
- 힙 메모리(Heap Memory)
  - 동적 메모리 할당에 사용되는 자료구조인 힙을 이용하여 객체 또는 동적 데이터 저장
  - v8 엔진 내부에서 가장 큰 공간을 차지
  - 가비지 컬렉션이 발생하는 곳
- 콜 스택(Call Stack)
  - 자바스크립트 코드가 실행되며 생성되는 실행 컨텍스트를 저장하는 자료구조
  - Javascript는 싱글 스레드 기반의 언어로, 콜 스택이 하나이기 때문에 한 번에 한 작업만 실행
  - 후입선출(LIFO)의 구조
    - > 함수를 실행하면 해당 함수가 콜 스택의 가장 상단에 위치
    - > 함수의 실행이 끝나면 해당 함수가 콜 스택의 가장 상단에 위치하고 있기 때문에 바로 제거됨
  - 스택 프레임(Stack Frame)
    - > 콜 스택에 쌓이는 데이터
    - > 순서대로 쌓여 스택 추적(Stack trace) 가능
  - 스택 오버플로(Stack Overflow)
    - > 자료구조 특성상 크기에 제한이 있기 때문에 한정된 메모리 공간을 넘어버리게 되면 발생하는 에러
    - > 콜 스택 내부의 동일한 스택 프레임이 예상치 못한 수로 쌓일 때 일어남

#### 자료 저장소
- 모든 자원을 하드 디스크에 저장할 필요가 있기 때문에 존재
- 영구적인 저장소인 로컬 스토리지(localStorage)와 임시적인 저장소인 세션스토리지(sessionStorage)를 따로 두어 데이터의 지속성을 구분할 구분할 수 있어 응용 환경에 맞는 선택이 가능
##### 특징
- 웹 브라우저가 직접 데이터를 저장할 수 있음
  - 모든 정보는 서버로 전송되지 않고 오직 클라이언트에만 데이터가 존재함
  - 네트워크 트래픽 비용이 줄어듦
- 오리진(origin)마다 단 하나씩만 존재
  - 하나의 오리진에 속하는 모든 웹 페이지는 같은 데이터를 저장하기 때문에 같은 데이터에 접근할 수 있음
